# 撚二科產銷戰情室開發手冊 (GEMINI.md)

本文件定義本專案之核心邏輯、數據處理機制及自動化規範，確保系統之準確性與維護性。

## 1. 數據流與整合架構 (Data Flow)

系統整合三方數據源，進行動態分析：
1. **排產計畫 (May表)**：`DTYMay26*.xlsx` 之第一分頁，定義「目標」。
2. **繳庫庫存 (庫表)**：`DTYMay26*.xlsx` 之第二分頁，定義「已完成實績」。
3. **現場實產 (實產表)**：`115年04月撚二科生產資訊.xlsx` 之 Index 1 分頁，定義「現場當下速度與狀態」。

## 2. 核心數據邏輯 (Core Logic)

### A. 批號匹配規則 (`simplify_id`)
* 移除 `FD` 前綴 (如 `FD2F` -> `2F`)。
* 移除字尾 `K` (通常不影響等級判定)。
* 統一轉大寫並去除前後空格。

### B. 側邊序列匹配規則 (Side Sequence Rule) - **核心規則**
由於實產表不直接標註 A/B 側，系統依據「資料出現序位」進行判定：
1. **AFK機台 (A07-24)**：第 1 筆出現 = **A側**，第 2 筆出現 = **B側**。
2. **TMT/33J機台 (T01-06, J07-09)**：第 1 筆出現 = **L側**，第 2 筆出現 = **R側**。
3. **匹配機制**：系統讀取計畫表的 `Mark` 欄位 (A/B/L/R)，並精確對接上述判定之序位數據。這決定了「實際批號」與「現場狀態文字」的歸屬。

### C. 產量分攤與均分機制 (Divisor Logic)
為解決機台分側計畫導致數據倍計之問題：
1. **現場總產量**：系統先加總實產表中該實體機台的所有 T/D 數值。
2. **計畫行數均分**：`實際 T/D (單行)` = `現場總 T/D` / `該機台在計畫表中的總列數`。
3. **庫存分攤**：`已繳庫量 (單行)` = `該批號總庫存` / `該批號在計畫中出現的總次數`。

### D. 現場狀態文字擷取
* **觸發條件**：當現場回報產量為 0 時。
* **來源**：優先抓取 Index 47 (實際產量T/日) 或 Index 26 (AA1 欄位) 內的非數值文字。
* **顯示**：標記於網頁「機台」欄位下方，以紅色粗體顯示 (如：`● 改紡`、`● 了機`)。

### E. 系統自動推估完工日 (System Forecast)
* **公式**：`實產表日期 + (計畫欠量 / (分攤後的實際 T/D * 1000))`。
* **智慧標籤**：
    * 繳庫量 ≥ 目標量時，顯示「**達標**」(綠色)。
    * 推估日期 > 7 天，顯示「**預計完工: MM/DD**」(橘色警告)。
    * 正常範圍內，顯示「**預計完工: MM/DD**」(綠色)。

## 3. 過濾與篩選規則 (Filtering)

### 機台排除清單
* 排除 `V`, `S2` 開頭機台。
* 排除 `庫存不排產`, `待排產`, `機台` 等文字行。
* **新增排除**：完全過濾 `M01-M08` 與 `S01-S02` 機台，不納入計算。

### 工程師篩選
* **吳**：A10-12, A16-18, A22-24, T01-03, J08。
* **羅**：A7-9, A13-15, A19-21, T04-06, J07。

## 4. 自動化維護
* **更新機制**：執行 `更新生產進度表.bat`。
* **處理流程**：`Git Pull` -> `Python 數據整合` -> `生成 HTML` -> `Git Push`。

## 5. 檔案結構
* `generate_dashboard.py`: 核心處理邏輯 (Python)。
* `production_dashboard.html`: 戰情室網頁產出 (HTML/JS)。
* `GEMINI.md`: 專案逻辑與開發指南。
