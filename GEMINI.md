# 撚二科產銷戰情室開發手冊 (GEMINI.md)

本文件定義本專案之核心邏輯、數據處理機制及自動化規範，確保系統之準確性與維護性。

## 1. 數據流與整合架構 (Data Flow)

系統整合三方數據源，進行動態分析：
1. **排產計畫 (May表)**：`DTYMay26*.xlsx` 之第一分頁，定義「目標」。
2. **繳庫庫存 (庫表)**：`DTYMay26*.xlsx` 之第二分頁，定義「已完成實績」。
3. **現場實產 (實產表)**：`115年04月撚二科生產資訊.xlsx` 之 Index 1 分頁，定義「現場當下速度」。

## 2. 核心數據邏輯 (Core Logic)

### A. 批號匹配規則 (`simplify_id`)
* 移除 `FD` 前綴 (如 `FD2F` -> `2F`)。
* 移除字尾 `K` (通常不影響等級判定)。
* 統一轉大寫並去除前後空格。

### B. 分側分配機制 (Divisor Logic) - **重要**
為解決同一機台(如 A09)因 A/B 側或不同時段產生多行計畫，導致產量虛報之問題：
1. **計畫出現次數**：系統會自動統計機台(如 A09)在計畫表中出現的總列數。
2. **產量均分**：`實際 T/D (單行)` = `現場回報總 T/D` / `計畫總列數`。
3. **庫存均分**：`已繳庫量 (單行)` = `該批號總庫存` / `該批號在計畫中出現的總次數`。
4. **顯示優化**：讀取 D 欄位(Mark)，若有 A/B 標註，則機台名稱顯示為 `A09 (A)` 或 `A09 (B)`。

### C. 產量抓取規則 (T/D Capture)
針對實產表 (Sheet 1) 之抓取優先序：
1. **優先**：Index 47 (實際產量T/日)。
2. **備援**：若 Index 47 為空(停機/改紡)，則抓取 Index 26 (AA1 欄位)。
3. **禁忌**：禁止使用 Index 25 (落紗數) 作為產量計算依據。

### D. 系統自動推估完工日 (System Forecast)
捨棄人工填寫的完工紀錄，改採科學推算：
* **公式**：`資料日期 + (計畫欠量 / (分攤後的實際 T/D * 1000))`。
* **標籤顯示**：
    * 繳庫量 ≥ 目標量時，顯示「**達標**」(綠色)。
    * 推估日期超過 7 天未完工，顯示「**預計完工: MM/DD**」(橘色警告)。
    * 正常範圍內，顯示「**預計完工: MM/DD**」(綠色)。

## 3. 網頁 UI 規範 (Web UI)

### 頂部摘要區塊
* **排產總目標**：計畫總噸數。
* **今日現場產量**：實體機台現場 T/D 總和 (不重複計算)。
* **運行中機台**：現場有回報數據的實體機台總數。

### 表格呈現
* **機台**：顯示清洗後機台號與 A/B 側標記。
* **批號 (計畫/現場)**：顯示計畫批號，並以「燈號」標示與現場批號是否一致 (🟢/🟡/⚪)。
* **生產效率**：顯示「計 vs 現」的 T/D 差異，並以 ▲/▼ 標示效率落差。
* **需求量**：有效繳庫量扣除計畫量 (正數綠字、負數紅字)。

## 4. 自動化與維護
* **自動更新**：透過 `更新生產進度表.bat` 觸發 Python 腳本並同步至 GitHub Pages。
