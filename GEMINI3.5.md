# 撚二科生產戰情室 3.5 旗艦版技術手冊 (GEMINI3.5.md)

本文件定義 3.5 「PREMIUM 旗艦版」的核心架構與運算邏輯。此版本實踐了「事實導向」的數據對接模式，並結合產銷計畫達成率與庫存智慧合併。

## 1. 系統架構 (System Architecture)
系統採用解耦設計，確保資料分析與視覺呈現徹底分離：
1.  **`RealityLogAnalyzer` (後端引擎)**：位於 `reality_analyzer.py`。負責讀取「生產資訊」、「DTY-LISA 庫存」、「絲八科 POY 庫存」與「產銷計畫表」，執行多維度的資料標準化與對接。
2.  **`generate_v3_html` (渲染層)**：位於 `generate_dashboard_v3.py`。負責 Premium UI 渲染、RWD 佈局切換、以及前端互動分析邏輯。

## 2. 核心運算邏輯 (Core Logic)

### A. 批號標準化與對接 (Standardization & Join)
為了達成跨檔案的精確匹配，系統實施了兩種標準化規則：
1.  **`standardize_id` (計畫對接用)**：
    *   移除 `FD`, `FP` 與空格。
    *   **保留 `LIKE` 關鍵字**：確保預排批號與正式批號不衝突。
    *   **修剪結尾字母**：使用正則 `(?<=\d)[A-Z]+$` 移除數字後的裝飾碼（如 `2F42S2K` -> `2F42S2`）。
2.  **`get_inventory_key` (庫存對接用)**：
    *   **移除 A/AX 字首差異**：移除 `2F`, `2G`, `8G` 前綴，僅保留核心編碼（如 `2F29B1` -> `29B1`）。
    *   **A+AX 合併**：此規則容許事實端的 A 級批號自動抓取庫存端對應的核心編碼重量，實現 A 級與 AX 級庫存的合併計算。

### B. 進度條計算 (Progress Bar - Fixed Denominator)
*   **分母固定法**：以「產銷表」中該批號在本月的整月計畫總重量作為分母。
*   **對接優先級**：
    1. 優先匹配「同機台 + 同批號」。
    2. 若失敗，則搜尋「全場」是否有相同批號的計畫目標（解決現場機台動態調度問題）。
*   **視覺回饋**：
    *   達成率 < 80%：顯示 **藍色**。
    *   80% <= 達成率 < 100%：顯示 **橘色** (預警即將了機)。
    *   達成率 >= 100%：顯示 **綠色** (達標)。

### C. POY 跨機台分析 (POY Shared Analysis)
*   **全場日耗運算**：自動掃描所有運作中機台，加總使用同一 POY 批號的「平均日產 (T/D)」。
*   **支撐天數**：`POY 庫存 (KG) / (全場總日耗 (T) * 1000)`，提供精確的斷料預警。

## 3. RWD 3.0 佈局規範
*   **斷點 (Breakpoint)**：`1200px`。
*   **卡片模式 (Card Mode)**：在手機端將表格轉化為 flex 卡片，內容標籤透過 `td::before` 偽元素生成，確保在窄屏下資訊依然條理清晰。
*   **縮減尺寸**：頂部 KPI 採用緊湊型設計，適合在現場大型顯示器或行動裝置上長時間監控。
